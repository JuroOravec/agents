# PRD Review Output Comparison: output_sonnet46 vs output_composer15

Both outputs were generated by running `src/crews/prd-review.ts` with either **Anthropic Sonnet 4.6** (OpenRouter) or **Cursor Composer 1.5**.

**Conclusion:** `output_sonnet46.md` is stronger for implementation; `output_composer15.md` is stronger for product/strategy context.

---

## Where output_sonnet46.md Shines

1. **Implementation-ready** — Concrete technical specs: `set -euo pipefail`, shebang, script location (`.cursor/bin/`), `755` permissions, explicit allowlist rules. Section 5 reads like a spec a developer could implement without guessing.

2. **Actionable acceptance criteria** — Section 9 has checkboxes (wrappers exist, pass `bash -n`, bare `gh` absent from allowlist, etc.). Composer15's success criteria are vaguer.

3. **Stronger security depth** — Tamper prevention, subcommand containment, supply chain expectations, and argument-injection risk are spelled out. Composer15 mentions trust and argument handling but at a lighter level.

4. **Governance** — The 6-step process for adding wrappers (proposal → security classification → script authoring → review → allowlist update → docs) is explicit. Composer15 has a looser maintenance model.

5. **Outstanding questions are tactical** — Things implementers need to resolve: PATH mechanism, `gh` version, error messaging, etc. Composer15's questions skew more strategic ("temporary workaround or long-term?", success metrics, Cursor roadmap alignment).

6. **More complete v1.0 scope** — Includes `gh_pr_status` (8 wrappers vs 7 in Composer15).

---

## Where output_composer15.md Shines

1. **Product framing** — Project classification, rollout/rollback, assumptions, open risks. Better for stakeholder buy-in and planning.

2. **Reusable "Definition of Safe"** — Four criteria that can be applied consistently to new subcommands. Sonnet46 defines safety by example and process, not by a formal taxonomy.

3. **Cross-platform** — Explicitly calls out Windows (Batch, PowerShell) and flexible install locations (`~/.local/bin/gh-wrappers/` or project `tools/`). Sonnet46 is bash/Unix-focused.

4. **More concise** — Easier to skim; less ceremony.

5. **Strategic questions** — "Temporary workaround or long-term?", quantified benefit, alignment with Cursor roadmap — useful for PM/product decisions.

---

## Bottom Line

- **If the goal is for an engineer to build this:** use `output_sonnet46.md`. It's closer to a spec they could implement with minimal ambiguity.
- **If the goal is product alignment and roadmap planning:** `output_composer15.md` is stronger on strategy, rollout, and risk — but would need implementation details (e.g. from sonnet46's Section 5) added before build-out.
